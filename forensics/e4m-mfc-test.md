# E4M 2.01 (Le Roux, 1999) — MFC fingerprint test

## What was tested

The Le Roux / Windows-C++-encryption wildcard reading has been argued in
popular press as a plausible Satoshi candidate. Two codebases attributable
to Le Roux exist in publicly accessible form:

- **TrueCrypt 7.1a** (2011, "TrueCrypt Foundation" team derivative of e4m):
  ruled out 2026-05-27 in commit `a36eb31`. Hungarian_C 0.1%, indent 91%
  tabs, composite z = −1.93 to −2.19 (the value shifts as new authors are
  added to the population and z-scores rebase; relative ranking is stable).
- **e4m 2.01** (1999, Le Roux's direct single-author work): tested here.

e4m is the **direct predecessor of TrueCrypt** — TrueCrypt v1.0 in 2004 was
a fork of e4m 2.0.2. Testing e4m directly closes the "pre-TrueCrypt Le Roux
might score differently" gap left by the 2026-05-27 TrueCrypt-only test.

Provenance, sha256, and pull reproducer at
[`code-corpus/e4m/SOURCE.md`](../code-corpus/e4m/SOURCE.md).

## Structural caveat: e4m is pure C

E4M 2.01 is implemented entirely in C (41 `.c` files + 52 `.h` files,
zero `.cpp` or `.hpp`). The Hungarian_C class-name pattern
(`class C[Capital]`) is structurally impossible in C — the language has no
`class` keyword. Le Roux's preferences on the C++-class-naming axis are
**invisible** in this corpus, regardless of what those preferences might
be.

This is the same gap as Back's Hashcash (also pure C), discussed in
[`forensics/nyt-april-2026-adam-back.md`](nyt-april-2026-adam-back.md).

The other two MFC fingerprint axes (space-indent ratio, line-comment
density) are not language-restricted and remain fully testable.

## Per-axis comparison

### Hungarian_C — structurally constrained to zero

e4m has zero `class C[Capital]` declarations because it has no classes at
all. The 0.1% rate measured is residual noise from C identifiers that
incidentally begin with `C[Capital]` (`CipherTextLen`, `Cmdline`, etc.) and
match the `^C[A-Z]` regex without being MFC class names. This is not
informative about Le Roux's MFC preferences.

The TrueCrypt 7.1a test was informative here: TrueCrypt 7.1a is C++ and
contains classes, but uses PascalCase-without-the-C-prefix
(`WizardPage`, `TrueCryptFactory`, `BaseCom`). If the TrueCrypt team
inherited Le Roux's preferences, those preferences include not using MFC
class naming.

### Indent style — tab vs space

e4m: 87% tabs / 13% spaces.
Satoshi: 0% tabs / 100% spaces.

e4m follows the Unix-on-Windows tab-indent convention common in 1990s
crypto-tools (compare TrueCrypt at 91% tabs, Finney's RPOW at 88% tabs).
This is the OPPOSITE of Satoshi's pure space-indent style. Le Roux's actual
preference is observable on this axis and is firmly anti-Satoshi.

### Comment density per KLOC

e4m: 80.7 block + 1.4 line per KLOC.
Satoshi: 1.0 block + 105.1 line per KLOC.

The ratio is **completely inverted**. e4m is 98.3% block comments / 1.7%
line comments by count (1.4 / 82.1). Satoshi is 99.1% line comments / 0.9%
block comments (105.1 / 106.1). This is a sharper inversion than any other
candidate; e4m's commenting style is more block-heavy than even
TrueCrypt's (22.5 block / 34.6 line).

The block-comment-heavy style is consistent with 1990s C convention (the
`//` line-comment form was retrofitted into ISO C99 and adopted slowly
afterward). e4m predates this transition; its block-only style is era-
appropriate for 1998–1999 C code. Satoshi's pervasive `//`-style line
comments are era-appropriate for 2008 C++ code. This single axis alone
distinguishes Le Roux's actual style from Satoshi's by a structural
margin.

### Burrows' Delta on code function-words

e4m: Δ = 0.944 from Satoshi (rank 6 of 9 non-Satoshi candidates).

The code function-word vocabulary (i, j, n, tmp, ret, buf, etc.) is the
closest analogue to prose function-words. e4m's identifier vocabulary is
in the mid-pack, neither closest nor furthest. The result is consistent
with e4m being a Windows-NT-driver-and-GUI codebase whose low-level
vocabulary (file handles, NT IRP packets, GUI state) overlaps moderately
with Bitcoin 0.1.x's low-level vocabulary (file handles, mutex state,
serialization buffers).

## Composite z-score interpretation

e4m composite: **−2.52** (third-from-last; only Finney is lower at −2.53).

The composite is overwhelmingly negative because e4m fails on the two
non-constrained axes simultaneously: tab-dominant indent AND
block-comment-dominant style. The Hungarian_C axis contributes nothing
diagnostic (structural zero), but the OTHER two axes pull the composite
firmly below mid-pack.

Satoshi composite: +3.67. The 6.2 z-score gap is the quantitative measure
of "e4m is not Satoshi's house style on the two axes where Le Roux's
preferences are observable."

## What this rules in / rules out

### Rules out: Le Roux as Satoshi (now via two independent codebases)

The Le Roux wildcard reading is now ruled out via **two independent
codebases**:

1. **TrueCrypt 7.1a** (2011, team derivative): composite z = −1.93.
2. **e4m 2.01** (1999, direct single-author): composite z = −2.52.

Both fail on every testable MFC fingerprint axis. The e4m result is the
more probative of the two because it tests Le Roux's actual single-author
style, free of the TrueCrypt-team obfuscation. The two-codebase agreement
removes the "but TrueCrypt was a team codebase that diverged from Le
Roux's personal style" objection.

The Le Roux ruling now requires the same defense as ruling out any other
candidate: a separately-attributable corpus of Satoshi-styled code that
Le Roux is documented to have authored. No such corpus is currently in
evidence.

### Rules in: pure-C candidates cannot be ruled in or out on Hungarian_C alone

Back's Hashcash and Le Roux's e4m are both pure C. Neither can produce
`class C[Capital]` declarations regardless of the author's MFC preferences.
The Hungarian_C axis is **structurally void** for these candidates, and
the ruling on Le Roux (and on Back, in the NYT April 2026 discussion at
`forensics/nyt-april-2026-adam-back.md`) rests on the OTHER two axes —
indent style and line-comment density — both of which fail for both
candidates.

This matters methodologically: a Hungarian_C-only test would have been
**uninformative** about Le Roux. The composite z-score's two-of-three
testable axes is what makes the e4m ruling possible.

### Does not rule out

- **Le Roux had other Windows-C++ MFC training not represented in e4m.**
  e4m is the cleanest single-author Le Roux artifact available, but it is
  pure C. If Le Roux wrote MFC-styled C++ in commercial roles not
  represented in this archive, that work could in principle match
  Satoshi. We do not have such a corpus to test.
- **The "1990s Windows-C++ MFC-era cypherpunk-adjacent" wildcard space
  more broadly.** PGP 6.5 (2026-05-27) and TrueCrypt 7.1a + e4m 2.01
  (this analysis) are three tests of plausible candidates from that
  space. None match Satoshi at the composite level. The space is not
  exhaustively tested — Microsoft-internal projects, game studios, ATL
  /COM-heavy enterprise apps from the period would all score higher on
  Hungarian_C and might score similarly on space-indent and line-comment
  density. The MFC fingerprint narrows Satoshi to "Windows-C++ developer
  with MFC training," not "any specific cypherpunk-adjacent team."

## Limitations

1. **e4m 2.01 is the available version; 2.02a (the final) is not in this
   archive.** The 6-month delta between 2.01 (Sept 1999) and 2.02a (2000)
   is unlikely to shift the stylistic profile materially, but the
   replication's strict reading is "Le Roux's 2.01-snapshot style" not
   "Le Roux's last-public-snapshot style."
2. **e4m predates the `//`-as-default-line-comment convention.** The
   block-comment-heavy style is era-appropriate for 1998–1999 C code and
   may not reflect Le Roux's later (2000s, 2010s) preference. We have no
   later sole-authored Le Roux artifacts.
3. **No commit-history data.** e4m was distributed as versioned zips, not
   VCS. We cannot extract per-commit authorship signals; the analysis is
   purely codebase-aggregate.
