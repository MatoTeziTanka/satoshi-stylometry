# PGP for Windows 6.5.1i — MFC fingerprint test

## What was tested

The single most discriminating code-style feature in Satoshi's Bitcoin 0.1.x
codebase is the Microsoft Foundation Classes (MFC) Hungarian C-prefix class
naming convention (`CTransaction`, `CBlock`, `CKey`, `CCriticalSection`,
`CBlockIndex`) at 5.4–9.6% of identifiers. The five named candidates
(Back, Finney, Szabo, Dai, Sassaman) score 0.1–0.2% on this axis — a 30× to
60× separation. The Le Roux / TrueCrypt 7.1a wildcard expansion
(2026-05-27, commit `a36eb31`) was ruled out at the same threshold:
TrueCrypt scores 0.1%.

The remaining plausible wildcard for a Windows-C++ MFC-era codebase from
Satoshi's community is **PGP for Windows 6.x** — Network Associates' freeware
PGP release of 1999, the canonical cypherpunk-adjacent Windows-C++ codebase
of that era. The PGP team included Phil Zimmermann, Will Price, Jon Callas,
Dave Del Torto, and a number of other identified contributors. This test
adds the PGP 6.5.1i source as a new code-corpus author and re-runs the
existing pipeline.

Source provenance, license caveat, sha256 of the archive, and pull
reproducer are documented at
[`code-corpus/pgp-6.5/SOURCE.md`](../code-corpus/pgp-6.5/SOURCE.md).

## Per-axis comparison

### Hungarian_C (`class C[Capital]` pattern) — % of identifiers

PGP 6.5 is the **only non-Satoshi codebase with non-trivial Hungarian_C usage.**
It scores 0.8% — 4× to 8× higher than every other candidate (0.1–0.2%), but
still 8× to 12× **below** Satoshi (6.4–9.6%).

Concrete count: 194 `class C[Capital]` declarations spread across 567,408
LOC. Examples observed: `CMsgType`, `CExtImpl`, `CDecrypt`, `CWaitCursor`,
`CPGPClassFactory`, `CShellExt`, `CMessageQueue`, `CPriorityQueue`,
`CControlThread`, `CCoderRegistry`, `CPFWindow`, `CSoundOutput`,
`CCounterEncryptor`, `CPipe`, `CLevelMeter`, `CPFPacketsOut`. These are
recognizably MFC-style — most are wrappers around COM interfaces, Win32
controls, or async-pattern primitives, which is the canonical MFC use case.

By rate, however, PGP 6.5's MFC usage is **incidental** rather than
**pervasive**. Across 909,481 identifiers in the codebase, only ~7,300
match the `C[Capital]` pattern (0.8% — versus Satoshi's ~1,580 of 24,669,
a 6.4% rate that is structurally different from "the codebase happens to
include some MFC components").

### Indent style — tab vs space

PGP 6.5: 71% tabs / 29% spaces.
Satoshi: 0% tabs / 100% spaces.

PGP 6.5 follows the **Unix-on-Windows** indent convention (tabs dominant),
which is what the rest of the cypherpunk Windows-C++ ecosystem used. Satoshi
is the only codebase in this analysis that is purely space-indented. This
is the second axis on which PGP 6.5 diverges from Satoshi.

### Comment density per KLOC

PGP 6.5: 67.4 block comments + 45.4 line comments per KLOC.
Satoshi: 1.0 block comments + 105.1 line comments per KLOC.

Both codebases are heavily commented, but the **balance is inverted**. PGP's
comments are predominantly C-style `/* ... */` block comments (the dominant
C++ commenting convention in the 1990s). Satoshi's are predominantly
`//`-style line comments (the dominant C++ commenting convention from
roughly 2000 onward, after `//` was retrofitted into ISO C99 and adopted
universally). This is a third independent axis on which PGP 6.5 looks
older-stylistically than Satoshi.

### Burrows' Delta on code function-words

PGP 6.5: Δ = 1.1055 from Satoshi — **further** than TrueCrypt's Δ = 0.8934.

The code function-word vocabulary (i, j, n, tmp, ret, buf, etc.) is the
closest analogue to prose function-words for Burrows' Delta on source code.
PGP 6.5 is not close to Satoshi on this axis; the result is consistent with
PGP being a 1990s C-heavy codebase while Satoshi's Bitcoin 0.1.x is a
2000s C++-with-MFC-conventions codebase.

## Composite z-score interpretation

The composite z = `z(Hungarian_C) + z(space_ratio) + z(line_comment)`
collapses the three discriminating axes into one number per author. The
intuition: a codebase that matches Satoshi simultaneously on all three
axes scores high positive; a codebase that matches on one axis but
diverges on others gets cancelled out.

PGP 6.5 composite: **−1.27.** This is negative, mid-pack, and **below
Sassaman** (−1.17). The Hungarian_C bump (+0.19 vs Sassaman's +0.0 in raw
percentage) is overwhelmed by PGP's tab-indent and block-comment-heavy
style.

Satoshi composite: +3.35 (combined corpus), +3.36 (v0.1.0), +4.92
(November 2008 pre-release). The 4–6 z-score gap from PGP 6.5 is the
quantitative measure of "PGP 6.5 is not Satoshi's house style."

## What this rules in / rules out

### Rules out: PGP team house style as Satoshi's signature

The composite z-score discriminates PGP 6.5 from Satoshi at the same
order of magnitude as it discriminates Sassaman, Back, Finney, Dai, and
TrueCrypt. The combined fingerprint (high Hungarian_C + spaces + line
comments) is not present in PGP 6.5. **The "Satoshi was a PGP-team
contributor" reading is not supported by this stylometric signature.**
Individual PGP contributors (Will Price, Phil Zimmermann, etc.) cannot
be ruled in or out on this basis — the analysis tests the codebase's
house style, not the style of any single contributor's commits, which
we cannot decompose from this archive.

### Rules in: composite z-score discriminates better than Hungarian_C alone

PGP 6.5's 0.8% Hungarian_C rate is the highest non-Satoshi result on
record — 4× to 8× the named candidates. A single-axis test thresholded
at "0.5% or higher" would have produced a **false positive** on PGP 6.5.
The composite z-score correctly places PGP 6.5 below Sassaman (which has
weaker Hungarian_C but a friendlier space-indent ratio). This validates
the methodological choice in commit `5289408` to formalize the
fingerprint as a three-axis composite rather than a single Hungarian_C
threshold. **The composite z-score is doing the work the single-axis
test could not.**

### Strengthens: Satoshi's fingerprint is structurally MFC-house-style, not incidental

PGP 6.5 contains MFC components (194 `C[Capital]` class declarations)
because the codebase has COM and Win32-control wrappers that follow MFC
conventions out of necessity (MFC requires the naming on inherited
classes). Satoshi's codebase exhibits the same convention at 8–12× the
rate, applied across most class declarations regardless of inheritance.
**Satoshi's codebase is written in MFC house style throughout; PGP 6.5
is a non-MFC codebase that includes MFC components.** This is a
structural difference, not a degree difference.

### Does not rule out

- **The "Satoshi learned MFC at a non-cypherpunk Windows-C++ shop"
  reading.** PGP 6.5 is one team's house style; it is not the universal
  template for 1990s–2000s Windows C++. Microsoft-internal projects,
  game studios, ATL/COM-heavy enterprise applications, and most
  Visual Studio sample code from the period would all score higher on
  Hungarian_C than PGP 6.5 — possibly comparably to Satoshi. The MFC
  fingerprint narrows Satoshi to "Windows-C++ developer with MFC
  training," not "PGP team contributor specifically."
- **Specific individual contributors at PGP.** The analysis tests
  codebase house style. A single contributor's MFC-styled commits would
  be diluted across the 567,408-LOC team codebase. We do not have
  per-author commit data for PGP 6.5.

## Limitations

1. **PGP 6.5 is a team codebase, not single-author.** The composite z
   measures the team's aggregate style. A single MFC-fluent contributor
   on the PGP team could have produced Satoshi-matching code without
   showing up in this aggregate.
2. **Pre-MFC C++ practice was not standardized.** "MFC house style" is
   defined here by the three observed Satoshi-specific axes
   (Hungarian_C + space-indent + line-comments). A different fingerprint
   definition might rank PGP 6.5 differently. The three-axis composite
   is post-hoc to Satoshi's observed values.
3. **No e4m 2.0.1 in this analysis.** Le Roux's pre-TrueCrypt direct
   work (1998–2000) is the remaining untested wildcard. e4m is older
   than PGP 6.5 by ~1 year and is documented as MFC-styled in
   secondary sources. A subsequent expansion to e4m would close that
   gap.
